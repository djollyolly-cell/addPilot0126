## Summary

D2a replaced the disabled scheduler-transport `recordRateLimit` producers (`if (false && hasData) ctx.scheduler.runAfter(0, ...)`) with direct awaited `ctx.runMutation` calls from the four `vkApi.ts` action sites. `recordRateLimit` mutation predicate restored as `statusCode === 429` only. `callMtApi.onResponse` contract changed to `void | Promise<void>` with one-row-per-logical-call guard inside the retry loop.

The change closes the scheduled-jobs amplification path that contributed to the 2026-05-04/05 incident (`recordRateLimit` was the largest pending-version producer at `248,692` rows in the initial backlog snapshot). Telemetry for VK 429 events is now restored without the scheduler as transport.

## Caveat — stdout-side 429 counter is not authoritative

The design's acceptance criterion `vkApiLimits rows ≤ observed real 429` was intended to cross-check the Convex-side row count against backend stdout 429 logs. In this runtime:

- Refined grep for actual HTTP 429 / `Too Many Requests` in stdout: 1 hit
- Broad grep including substring `429` / `Too Many`: 67 hits, but mostly noise (e.g., `latency 494.429ms`, `response_size: Some(...429...)`)
- Neither number cleanly matches the Convex-side count of 10

**Authoritative verification source for the predicate is `vkApiLimits` itself**, not stdout: 10 rows, 100% `statusCode: 429`, 0% `statusCode: 200`. This proves only real 429 responses created rows, and no 200 responses leaked. The old scheduler-transport path also did not revive (0 new `_scheduled_functions` `recordRateLimit` entries throughout the window).

Backend stdout's role is the rollback-pattern check (`Too many concurrent` / `Transient error` / `TOKEN_EXPIRED` / `syncBatchV2 account failed` / `adminAlerts.notify`), all of which were 0.

For D2b (header sampling) or future D2 phases, do not rely on stdout 429-count as a closure gate. Use `vkApiLimits` table itself with the same `statusCode` filter for predicate verification.

Worst-case overlap unchanged from post-throughput baseline:

```
Token refresh worst-case: 6 V8 action slots
UZ worst-case:            6 V8 action slots
Sync current profile:     2 V8 action slots
Total:                   14 / 16 V8 action slots
Headroom:                 2 V8 action slots
```

D2a adds at most one awaited mutation round-trip per recorded 429 event. With 10 events over 3 hours, the latency cost is negligible. Any extension beyond `429`-only (D2b sampling on 200 responses, D2c aggregated state, or `vk-throttling-probe` restore) must revisit V8 slot math against this 2-slot headroom.
