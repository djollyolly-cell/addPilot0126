# Проблемы правил УЗ (uz_budget_manage) — Анализ 03.04.2026

## Проблема 1: Кампании из новых кабинетов не сохраняются при редактировании правила

### Суть
1. Пользователь создаёт правило УЗ с кампаниями из 2 кабинетов
2. Позже подключает ещё 3 кабинета
3. Открывает редактирование правила, добавляет кампании из новых кабинетов
4. Сохраняет — но новые кампании **не попадают** в `targetCampaignIds`

### Кейс: Ольга Чистякова (138644936@vk.com)
- Правило "Управление бюджетом" — targetAccountIds: 5 аккаунтов
- targetCampaignIds: 41 кампания, но **все** из первых 2 аккаунтов (Саратов, Уфа)
- 3 агентских аккаунта (Челябинск, BVC, СТРАННИК) — 0 кампаний в правиле
- Результат: бюджеты этих 3 аккаунтов не управляются, кампании стопятся по бюджету

### Корневая причина

**`convex/rules.ts:226-354`** — мутация `update`:
- Строки 344-346: принимает `targetCampaignIds` без валидации
- В `create` (строка 72-74) есть проверка: `if (targetCampaignIds.length === 0) throw Error`
- В `update` эта проверка **отсутствует** — можно сохранить правило УЗ с пустым списком кампаний

**`src/components/TargetTreeSelector.tsx:83-210`** — компонент `UzAccountNode`:
- При редактировании правила начальное состояние берётся из `existingRule?.targetCampaignIds`
- При добавлении кампаний из нового аккаунта возможна проблема со стейтом: toggle на уровне аккаунта может не корректно обновлять общий массив `selectedCampaignIds`

## Проблема 2: Одна кампания в нескольких правилах УЗ = двойное увеличение бюджета

### Суть
1. Создаётся правило УЗ-1, добавляются кампании A, B, C
2. Создаётся правило УЗ-2, добавляются кампании B, C, D
3. Кампании B и C попадают в оба правила
4. При срабатывании обоих правил — бюджет увеличивается **дважды**

### Корневая причина

**`convex/ruleEngine.ts:1673-1864`** — `checkUzBudgetRules`:
- Строки 1708-1719: фильтрация по `targetIds.includes(String(c.id))`
- Каждое правило обрабатывается **независимо** — нет проверки, обрабатывалась ли кампания другим правилом
- `hasRecentBudgetIncrease` (дедупликация) работает **только внутри одного правила**, не между правилами

**`convex/rules.ts`** — мутации `create` и `update`:
- Нет проверки на пересечение `targetCampaignIds` с другими активными правилами УЗ того же пользователя

---

## План исправлений

### Fix 1: Валидация в `update` (аналогично `create`)
**Файл:** `convex/rules.ts`, мутация `update`
**Что:** Добавить проверку `targetCampaignIds.length === 0` для uz_budget_manage при обновлении
**Риск:** Низкий — чистая валидация

### Fix 2: Дедупликация кампаний между правилами (create/update)
**Файл:** `convex/rules.ts`, мутации `create` и `update`
**Что:** При создании/обновлении правила УЗ проверять, не находятся ли выбранные кампании в других активных правилах УЗ этого пользователя. Если пересечение — вернуть ошибку с перечислением конфликтующих кампаний
**Риск:** Средний — нужно решить, блокировать или предупреждать. Рекомендация: блокировать (throw Error)

### Fix 3: UI — корректное обновление стейта TargetTreeSelector
**Файл:** `src/components/TargetTreeSelector.tsx`
**Что:** При редактировании правила убедиться, что кампании из всех аккаунтов (включая добавленные позже) корректно загружаются и сохраняются. Проверить `handleCampaignToggle` (строки 129-151) на корректность работы с кампаниями из разных аккаунтов
**Риск:** Средний — требует тестирования UI-стейта

### Fix 4: Runtime дедупликация в checkUzBudgetRules
**Файл:** `convex/ruleEngine.ts`, функция `checkUzBudgetRules`
**Что:** Перед обработкой правил — собрать Set всех кампаний, уже обработанных другими правилами. Пропускать кампанию, если она уже была увеличена в текущем цикле проверки
**Риск:** Низкий — дополнительная страховка на случай, если Fix 2 пропустит пересечение (например, правила создавались до внедрения проверки)
