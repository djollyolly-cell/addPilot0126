# Конструктор отчёта клиенту — Design Spec

**Дата:** 2026-04-19
**Статус:** Draft (ожидает ревью)
**Автор:** обсуждение с пользователем (brainstorming)

## Контекст и проблема

Пользователи AdPilot — таргетологи, которые ведут рекламу VK для своих клиентов (чаще всего агентства). Один из клиентов (Наталия, ниша «мебельные») запросила отчёт для своего клиента со следующими полями:

- День недели, дата
- Рекламный бюджет без НДС и с НДС
- Показы, переходы, CPC, CTR
- Число подписок на группу
- Число подписок на Senler
- Число стартов сообщений в группе VK
- Число номеров, которые оставляют клиенты в сообщениях или формах VK
- По номерам: имя, фамилия, ссылка на диалог

Сейчас таргетолог перебирает сообщения сообщества вручную — «держит спеца на это». Цель фичи — автоматизировать.

**Важно:** задача атрибуции сообщений per-banner (для rule engine) — **отдельная задача (Feature 2)**. Текущий дизайн её не решает. Клиенту-таргетологу нужны **агрегаты за день**, не per-banner.

## Цели

- Дать таргетологу self-serve конструктор отчёта с настраиваемыми полями и гранулярностью
- Автоматизировать извлечение данных из VK сообщений и Senler
- Выгрузка в Excel для отправки клиенту
- Сохранение шаблонов для повторного использования

- `phones_detail` автоматически включает `phones_count`

- При снятии галки с зависимости UI показывает предупреждение «включить автоматически?»

### Поведение

- Зависимости фильтров: при выборе кабинета dropdown кампаний фильтруется до кампаний этого кабинета
- Оценка времени: `accounts × communities × (daysInRange / 7) × 30 сек` (эмпирическая формула; уточним коэффициент после первых прогонов на реальных данных)
- Если оценка > 3 минут — плашка: «Для ускорения рекомендуем построить по одному кабинету за раз», кнопка «Применить для первого кабинета»
- Прогресс-бар: «Кабинет 2 из 3: Мебельный салон — загружаем диалоги 120/350»
- Предупреждение, если в выбранных кампаниях есть сообщества без `communityProfile`: «Для сообществ X, Y не подключён профиль. Сообщения/номера/Senler для них будут пустыми. Добавить профиль»
- Сортировка по дате: по умолчанию asc, переключение на desc
- Сортировка по другим колонкам — V2

### Пустое состояние
Если нет данных в периоде: «В выбранном периоде нет данных. Попробуйте расширить период или добавить кабинеты.»

## UI профилей сообществ

Новая карточка в `SettingsPage`.

### Список профилей

Карточка на каждый профиль с: аватар, имя сообщества, ссылка `vk.com/club{id}`, статус VK-токена, статус Senler, «Токен проверен N мин. назад», кнопки редактирования и удаления. Снизу: «Подключено: N / 50».

### Модалка «Добавить сообщество»

Трёхшаговая:

1. **Токен VK-сообщества (обязательно)**
   - Password input + ссылка «Как получить токен?»
   - Кнопка «Проверить» → `groups.getById` по токену → показать найденное сообщество
   - Ошибки: «Токен не подходит», «Нет прав messages», «Сообщество уже добавлено»
2. **API-ключ Senler (опционально)**
   - Password input + чекбокс «У меня нет Senler»
   - Кнопка «Проверить» → тестовый запрос
   - Можно пропустить
3. **Подтверждение**
   - Сводка: имя сообщества, статус VK, статус Senler
   - Кнопка «Сохранить»

### Ежедневный cron 04:00 UTC

Валидирует токены всех профилей:
- `groups.getById` с токеном VK → если 401/403 → `lastError`
- Senler API ping → если key invalid → частичный `lastError`
- Обновляет `lastValidatedAt`

Этот cron **не фетчит сообщения** — данные для отчёта берутся on-demand.

### Ручная ре-валидация
Кнопка «Проверить сейчас» в профиле — запускает ту же валидацию по требованию.

## Data fetching (on-demand)

Решено: **не храним** диалоги/сообщения/контакты — фетчим при каждом построении отчёта.

### Flow `buildReport(userId, filters, granularity, fields, dateFrom, dateTo)`

1. Валидация входа (минимум 1 кабинет, period ≤ 180 дней, fields ⊆ каталог)
2. **Ad metrics**: SELECT из существующей `metricsDaily` по `accountIds` + `date range`
3. **Community metrics** (если выбраны соответствующие поля):
   - Определяем `communityIds` из креативов указанных кампаний или из фильтра
   - Для каждого соответствующего `communityProfile`:
     - `messages.getConversations` — paginated, sorted by last_message DESC, пока последнее сообщение в диапазоне
     - Фильтруем диалоги по `firstMessageDate` в диапазоне
     - `messages.getHistory` с `rev=1`, `count=50` для каждого нового диалога — читаем **только входящие** (`from_id != -group_id`)
     - Regex phone extraction из `text` каждого сообщения
     - `users.get` (батчи по 100) для peer_ids → `{ first_name, last_name, photo_100 }`
     - Собираем в `phones: { phone, leftAt, peerId, firstName, lastName, dialogUrl }[]`
4. **Lead Ads** (если выбран `phones_detail` или `leads`):
   - `lead_ads/vkontakte/leads.json` за период
5. **Senler** (если `senler_subs`):
   - Senler API `/SubscribersFromGroup` за период
6. **Merge** по ключу `(date, [campaign], [group], [banner])` в зависимости от гранулярности
7. Итого: сумма/среднее для строки «Итого»
8. Return: `{ rows, totals, phonesDetail?, communitySummary? }`

### Агрегат `phones_count`

Считается по **дате оставления номера** (`leftAt.toISOString().slice(0, 10)`), а не по дате диалога. Это логичнее для воронки конверсий.

### Regex номеров

```typescript
const PHONE_REGEX = /(?:\+?(?:375|7|380)|8)[\s\-\(\)]?\d{2,3}[\s\-\(\)]?\d{3}[\s\-]?\d{2}[\s\-]?\d{2}/g;

function normalizePhone(raw: string): string {
  const digits = raw.replace(/\D/g, "");
  if (digits.startsWith("8") && digits.length === 11) return "+7" + digits.slice(1);
  if (digits.startsWith("375") || digits.startsWith("380") || digits.startsWith("7")) return "+" + digits;
  return "+" + digits;
}
```

Покрывает форматы: Россия (`+7`, `8`), Беларусь (`+375`), Украина (`+380`).

### Rate limits VK API
3 rps на токен. Между вызовами — `sleep(400ms)` + backoff при `error_code=6`.

### Обработка ошибок в процессе построения
- Токен протух посередине → отметить сообщество как «ошибка», пропустить, продолжить остальные
- В конце построения — вернуть `{ partialErrors: ["Сообщество X: токен истёк, обновите в настройках"] }`
- UI показывает warning-плашку с этими ошибками

### Где выполняется
Convex `action` (не query — долго, имеет right to fetch). UI — `useAction` + `useState` с прогрессом.

## Excel-экспорт

Библиотека `xlsx` (уже в dependencies). Экспорт на клиенте.

### Структура

**Имя файла:** `report_{dateFrom}_{dateTo}_{accountName}.xlsx`

**Листы:**

1. **«Сводка»** — метаданные: период, кабинеты, гранулярность, время построения, пользователь
2. **«Отчёт»** — основная таблица + строка «Итого» жирным внизу
3. **«Номера»** (если `phones_detail`) — одна строка на извлечение: дата, время, номер, имя, фамилия, ссылка на диалог, источник, контекст (кабинет/кампания/группа/баннер)
4. **«Сообщества»** (если используются `communityProfile`) — сводка: имя, VK ID, новых диалогов, номеров, подписок Senler

### Форматирование

- Денежные колонки: `#,##0.00 "₽"`
- Процентные: `0.00%`
- Даты: `DD.MM.YYYY`
- Заголовки: жирный, заливка `#E7EAF3`, выравнивание по центру
- Ссылки на диалог: кликабельные гиперссылки
- Auto-fit ширины колонок (max 50 символов)
- Freeze первой строки

### Не делаем в MVP
Графики, PivotTables, брендирование, PDF.

## CRUD шаблонов

### Действия
- **Создать (+ Новый)**: модалка с именем (max 60 chars, unique), сохраняет текущее состояние билдера
- **Сохранить (💾)**: обновляет выбранный шаблон (без изменения периода)
- **Загрузить**: клик в dropdown подставляет настройки шаблона
- **Удалить (🗑)**: подтверждение + удаление

### Лимит
10 шаблонов на пользователя. При попытке 11-го — ошибка «Лимит шаблонов».

### Устойчивость
Если в шаблоне удалённые `accountId`/`communityId` — тихо игнорируем, показываем warning «Часть кабинетов из шаблона больше недоступна».

### Валидация полей
При сохранении сравниваем `fields` с каталогом. Неизвестные поля — ошибка.

### Модалка «Управление шаблонами»
Простой список с именем, датой создания, кнопками «Открыть» и «Удалить».

### Переименование / копирование — V2.

## Открытые риски

1. **Производительность**: 5+ минут на отчёт 30+ дней может раздражать. Если так — ускоряем через V2 (шаг 3).
2. **Приватность**: не храним тексты/номера, но на время построения они в памяти и в ответе клиента. Клиент должен понимать, что номера идут через фронт (рекомендация: не логировать).
3. **VK API changes**: `ref`/`ref_source` могут измениться. В MVP мы их не используем — риск низкий.
4. **Senler rate limits**: не документированы явно, нужно узнать на практике.
5. **Вероятность rate limit VK**: при 3 rps и большом объёме может быть долго. Решаем sleep(400) + backoff.

## Технические открытые вопросы (решаем в планировании)

- **Как извлечь `community_id` из креатива кампании?** У myTarget креатива УЗ нужно разобрать поле `link_url` / `content` — оно может содержать `vk.com/club{id}` или `vk.me/{id}`. Точный путь определим при написании `clientReport.buildReport`.
- **Максимальный период отчёта** — предварительно ставим 180 дней (6 месяцев). Если VK API возвращает меньше истории — ограничение по факту.
- **Ответ action на 5+ минут**: Convex action имеет таймаут 10 минут. Если приближаемся к нему — нужна пагинация/чанкинг по кабинетам. В MVP пусть будет монолитным; если упираемся — в V2 разобьём.
- **Excel-экспорт большого объёма**: `xlsx` в браузере ограничен памятью. Для отчётов > 50K строк может потребоваться server-side генерация. В MVP считаем, что лимиты не достигаем.

## Метрики успеха MVP

- Наталия (или аналогичный пользователь) может за 1 клик «Применить шаблон» получить готовый Excel для клиента
- Время ручной работы таргетолога при подготовке отчёта снижается с «часов на спеца» до минут
- Новых отзывов «не хватает поля X» — не более 2 на протяжении месяца
