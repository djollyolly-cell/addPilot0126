## 1. Симптомы и диагноз

### Что наблюдалось
- WebSocket к `convex.aipilot.by` разрывается с кодом `1006` каждые 10–30 минут (>50 разрывов за сутки на одной вкладке `/admin`).
- На скриншоте от 2026-05-03 06:33 страница висит на спиннере — связано с перезагрузкой VPS в 06:29 (журнал hoster.by).

### Корневая причина (по диагностике Postgres)
- `max_wal_size = 1GB` (дефолт) при WAL-бёрстах ~5.9GB от autovacuum индексов → forced checkpoints каждые 20–30 секунд.
- `checkpoint_timeout = 5min` усугубляет: даже без бёрста чекпоинт раз в 5 минут пишет full-page writes.
- Latency Postgres скачет на каждом чекпоинте → Convex UDF в очереди → клиент не получает ack ping → Traefik/клиент закрывают WebSocket с `1006`.

### Дополнительные сигналы
- `xid_age ~96M` (48% от `autovacuum_freeze_max_age=200M`) — состояние умеренное, не критическое. Стоит наблюдать тренд после фикса: если возраст продолжит расти быстрее обычного, это отдельный сигнал по autovacuum, который копать независимо.
- `pg_stat_activity` сейчас не показывает зависших долгих транзакций.
- Свободного места на диске ~33GB.

## 2. План изменений

| Параметр | Сейчас | Целевое | Обоснование |
|---|---|---|---|
| `max_wal_size` | 1GB | 8GB | Поглощать WAL-бёрсты от autovacuum (5.9GB) одним чекпоинтом. 8GB ≈ 25% свободного места — безопасно. 32GB не ставим: при заполнении диска Postgres падает. |
| `checkpoint_timeout` | 5min | 30min | Меньше чекпоинтов → меньше full-page writes. Цена: ~30 сек длиннее recovery после краха. |
| `checkpoint_completion_target` | проверить | 0.9 | Размазать I/O чекпоинта по 90% интервала. На PG 14+ дефолт уже 0.9. |
| `wal_compression` | проверить | on | Сжимает WAL в 2–4 раза, особенно от autovacuum индексов. CPU-стоимость минимальна. |

Все параметры — `sighup`. **Рестарт Postgres не нужен.**

### ШАГ 4. Верификация применения (КРИТИЧНО)

```sql
SELECT name, setting, unit, source, sourcefile, sourceline, pending_restart
FROM pg_settings
WHERE name IN (
  'max_wal_size',
  'checkpoint_timeout',
  'checkpoint_completion_target',
  'wal_compression'
);
```

**Главный критерий успеха — фактическое значение, а не источник:**

| Параметр | Ожидаемое `setting` | `unit` |
|---|---|---|
| `max_wal_size` | `8192` | `MB` |
| `checkpoint_timeout` | `1800` | `s` |
| `checkpoint_completion_target` | `0.9` | (нет) |
| `wal_compression` | `on` | (нет) |

**Условия успеха (для каждой из 4 строк):**

1. `setting` = ожидаемое значение из таблицы выше,
2. `pending_restart` = `false`.

Если оба выполнены — **всё применилось**, дальше Шаг 5 (ожидание).

**`sourcefile` — диагностический инструмент, не критерий успеха.**

Если `setting` совпадает с целевым — задача решена независимо от `sourcefile`. К `sourcefile` обращаемся только когда `setting` НЕ совпадает с ожидаемым, чтобы понять, **откуда пришло «неправильное» значение**:

| `setting` ≠ цель — что смотрим | Вывод |
|---|---|
| `sourcefile` оканчивается на `postgresql.auto.conf` | ALTER SYSTEM записал, но не перебил более приоритетный источник — крайне редкий случай, копать дальше |
| `sourcefile` оканчивается на `postgresql.conf` (основной) | Хардкод в основном конфиге перебивает `auto.conf` — править его напрямую через volume mount |
| `source = 'command line'`, `sourcefile = NULL` | Параметр приходит из Docker `command:` в compose. `ALTER SYSTEM` запись в `auto.conf` сделал, но command-line перебивает auto.conf по приоритету — фактическое значение не меняется. См. План B |
| `source = 'default'` | Значение не сохранилось — проверить права (нужен superuser) и что `pg_reload_conf()` вернул `t` |
| `pending_restart = true` | Параметр требует рестарта (для наших 4 такого быть не должно) |

**Важная оговорка:** различить `postgresql.conf` и `postgresql.auto.conf` можно **только по `sourcefile`**. `source = 'configuration file'` сам по себе ничего не говорит — оба файла отдают одинаковое значение этой колонки.

## 5. Откат (если что-то пошло не так)

Безопасно — никаких данных не трогаем:

```sql
ALTER SYSTEM RESET max_wal_size;
ALTER SYSTEM RESET checkpoint_timeout;
ALTER SYSTEM RESET checkpoint_completion_target;
ALTER SYSTEM RESET wal_compression;
SELECT pg_reload_conf();
```

Вернёт дефолты Postgres (1GB / 5min / 0.9 / off).

**Про состояние `postgresql.auto.conf`:** `ALTER SYSTEM SET` пишет в этот файл, `ALTER SYSTEM RESET` — удаляет соответствующую запись из него. Поэтому после отката `auto.conf` чист. Это важно, если параметр изначально приходил из command-line (План B): если потом флаг из compose уберут, в `auto.conf` не останется «забытых» наших значений, которые внезапно «всплывут».

## 6. Связь с WebSocket-разрывами

Гипотеза: чекпоинт-шторм каждые 20–30 сек → write latency Postgres скачет → Convex UDF в очереди → клиент не получает ack ping → WebSocket закрывается с `1006`.

Проверка после фикса (1–2 часа наблюдения на вкладке `/admin`):
- Если разрывы пропали → гипотеза подтверждена, причина одна.
- Если частота разрывов та же → причина в Traefik (idle timeout). Тогда отдельная задача: настроить `idleTimeout` в Dokploy → Settings → Traefik для `convex.aipilot.by` (поднять до `1h+` для WebSocket-роутеров).

- **xid_age** должен начать падать в течение суток после фикса — autovacuum пойдёт freeze-фаза без прерываний.

## 10. Сильный кандидат на фоновую latency: cache thrashing при `shared_buffers = 128 MB`

> **Дисклеймер:** этот раздел описывает **кандидата** на причину фоновой latency, не доказанную причину. Без `pg_stat_statements`, wait events или измерений query latency это пока гипотеза, основанная на сильном сигнале в bgwriter-метриках. Доказательство требует отдельного измерения.

### Полный `pg_stat_bgwriter` за 69 минут после reset

```
checkpoints_timed     =        0
checkpoints_req       =        3
buffers_checkpoint    =       72   (0.0008%)
buffers_clean         = 1,195,511   (11.8%)   ← bgwriter
maxwritten_clean      =   11,146   ← bgwriter упирался в лимит 11K раз
buffers_backend       = 8,912,217   (88.2%)   ← writes от backends
buffers_backend_fsync =        0   ← важно: fsync backends НЕ делали
buffers_alloc         = 21,196,928  (169 GB через 128 MB кэш)
```

### Правильная интерпретация (с поправкой по `buffers_backend_fsync`)

`buffers_backend = 14M, buffers_backend_fsync = 0` означает:

- Backends действительно делали `write()` syscalls для грязных страниц (быстрые, попадают в OS page cache).
- Но **синхронный fsync** при этом НЕ выполняли — это сделал бы счётчик `buffers_backend_fsync > 0`.
- Поэтому **запросы НЕ блокируются на fsync**. Записи асинхронные.

То есть **изначальная формулировка «backend синхронно ждёт fsync» была неверной**. Реальный путь latency другой.

### Реальный механизм latency — cache thrashing, не sync writes

Расчёт пропускной способности кэша:
- `buffers_alloc / 69 мин = 5,100 buffer allocations/sec` через 128 MB пул.
- `buffers_alloc / shared_buffers_pages = 21.2M / 16,384 ≈ 1,294` — каждый буфер пере-используется ~1300 раз/час.

При таком темпе данные вытесняются из `shared_buffers` **до того, как пригодятся** для следующего запроса. Это значит:

1. Запрос приходит → нужная страница НЕ в `shared_buffers` (была вытеснена) → **disk read** (или OS page cache hit).
2. Эти disk reads — **синхронные** с точки зрения запроса. Backend ждёт.
3. Latency запроса определяется тем, успел ли OS page cache закэшировать вытесненную страницу или придётся читать с физического диска.

Параллельно — но **не блокируя запросы**:
- Backends переписывают грязные буферы (асинхронно через write syscall).
- bgwriter постоянно упирается в лимит (`maxwritten_clean = 11,146`).
- Checkpointer фактически только ставит fsync-маркеры.

### Серьёзность

- БД `adpilot_prod` = **137 GB**.
- `shared_buffers = 128 MB` = **0.09% БД помещается в кэш**.
- 1,294 пере-использований буферов в час = тяжёлый cache thrashing.

### Чего ЭТО НЕ объясняет напрямую

- WS-разрывы 1006 — нет прямой связи. Cache thrashing влияет на **query latency**, но не на стабильность WebSocket-соединения.
- Конкретные пиковые проблемы UI — без `pg_stat_statements` нельзя сказать, какие именно запросы страдают.

### Что нужно для доказательства

Без этого — это сильный сигнал, не вердикт:
1. `pg_stat_statements`: top-20 запросов по `total_exec_time` и `mean_exec_time`.
2. `pg_buffercache` или `pg_stat_database.blks_hit / blks_read` для cache hit ratio (норма >99%, проблема <90%).
3. Wait events (`pg_stat_activity.wait_event`) во время пиков нагрузки.
4. P95/P99 latency запросов из приложения (Convex action timings).

### Расчёт серьёзности

- БД `adpilot_prod` = **137 GB**
- `shared_buffers = 128 MB` = **0.09% БД помещается в кэш**
- `buffers_alloc = 21M / 69 мин` = **5,100 аллокаций в секунду** через 128 MB кэш
- `maxwritten_clean = 11,146` = bgwriter за 69 мин 11K раз останавливался по `bgwriter_lru_maxpages` (дефолт 100)
- На каждые 100 страниц, что bgwriter записывает фоном, backends делают ~745 записей сами

### Cache thrashing

`buffers_alloc / shared_buffers_pages = 21,196,928 / 16,384 ≈ 1294`.
То есть каждый из 128 MB буферов **переписывается ~1300 раз в час**. Это полный thrashing: ничего не остаётся в кэше, каждый запрос идёт за данными на диск.

### Вывод

Чекпоинт-фикс убрал **пиковые латенси-всплески** (один из двух источников). Но **базовая** фоновая latency от backend-eviction осталась — её 88% всей dirty-page активности.

Если UI latency останется высокой после устранения других факторов — это **возможное объяснение медленных запросов**, но **не прямое объяснение WS 1006**. Связь cache thrashing с WebSocket-стабильностью отсутствует напрямую: thrashing влияет на query latency, а не на стабильность TCP/WS-соединения. Это значимая работа отдельным планом, не правкой одного параметра.

## 11. Следующий план (отдельная сессия)

**Цель:** убрать backend-eviction как доминирующий путь записи грязных страниц.

### Параметры к настройке

| Параметр | Сейчас | Целевое | Контекст | Risk |
|---|---|---|---|---|
| `shared_buffers` | 128 MB | **8 GB** (~17% RAM) | postmaster | требует рестарт Postgres (~10–30 сек downtime) |
| `effective_cache_size` | (default 4 GB) | **32 GB** (~70% RAM) | sighup | reload-able, безопасно |
| `bgwriter_lru_maxpages` | 100 | **500–1000** | sighup | reload-able |
| `bgwriter_delay` | 200 ms | **100 ms** | sighup | reload-able |
| `work_mem` | (default 4 MB) | **обсуждаемо** | session | требует анализа отдельных запросов |

### Порядок применения

1. Сначала reload-able (`effective_cache_size`, `bgwriter_*`) через `ALTER SYSTEM` без рестарта.
2. Замерить эффект (24 часа).
3. Только потом `shared_buffers` с рестартом — если эффект первого шага недостаточен.

### Чего НЕ делать сразу

- Не ставить `shared_buffers > 25% RAM` — эмпирическое правило для PostgreSQL: больше 25% не даёт пользы из-за двойной буферизации с OS page cache.
- Не трогать `work_mem` без понимания топ-N запросов по памяти.
- Не делать всё одной транзакцией — изменения постмастер-уровня требуют рестарта, reload-able параметры дают мгновенный сигнал.

## 14. Найдена основная причина WS-разрывов: Convex Isolate memory carry-over

### Анализ логов Convex backend за окно замера (07:13–07:43 UTC)

```
51 ERROR / 30 min
46 WARN  / 30 min
```

### Три типа сообщений

**1. `Restarting Isolate memory_carry_over: TooMuchMemoryCarryOver("60-64 MiB", "96 MiB")`** — V8 Isolate в Convex имеет лимит heap 96 MiB. Когда после GC между запросами остаётся >60 MiB — Convex принудительно рестартует Isolate. **Все WS-соединения через этот Isolate в этот момент рвутся.**

**2. `WebSocket protocol error: Connection reset without closing handshake: Client disconnected`** — это и есть `code 1006` со стороны клиента. Симптом, не причина.

**3. `Websocket ping/pong timeout: Client disconnected`** — Convex сам убивает соединение, не дождавшись pong. Появляется когда клиент busy (часто следствие Isolate restart).

### Конкретные функции, перегружающие Isolate

```
07:24:41  Action: vkApi.js:getMtBanners              (63.70 MiB carry-over)
07:25:32  UDF:    adAccounts.js:upsertCampaignsBatch (63.67 MiB)
07:25:49  Action: vkApi.js:getMtStatistics           (63.57 MiB)
07:36:39  Action: vkApi.js:getMtBanners              (60.25 MiB)
```

### Корреляция с разрывами

В логах видны **массовые синхронные разрывы**:

- `07:33:19` — **8 разрывов в одну секунду** = один Isolate restart убил 8 параллельных WS
- `07:38:27` — **7 разрывов в одну секунду** = ещё один restart
- `07:41:26` — **7 разрывов в одну секунду** = ещё один

В одной браузерной вкладке = 1 разрыв, но в логах сервера их 7–8 одновременно — другие пользователи в этом же Isolate. **Каждый Isolate restart = массовый WS-disconnect.**

### Исследование кода (по реальным файлам)

**`convex/syncMetrics.ts` line 938+** — в per-account worker одновременно живут (на лимите 96 MiB heap):

| Переменная | Источник | Размер |
|---|---|---|
| `banners` | `getMtBanners` (full объекты с textblocks, urls) | большой, линейно растёт |
| `bannerIds` | derived | мелкий |
| `stats` | `getMtStatistics` по всем bannerIds | большой |
| `leadCounts` | `getMtLeadCounts` | средний |
| `bannerCampaignMap` | Map | средний |
| `vkCampaigns` | `getCampaignsForAccount` | средний |
| `groupData` | Map | средний |
| `campaignTypeMap` | Map | средний |
| `fetchedAdPlans` | array | средний |
| `adPlanBudgets` | Map | мелкий |

**ВСЕ одновременно** в одном sync-worker'е. Реальная конкурентность: `convex/syncMetrics.ts:15` объявляет `WORKER_COUNT = 6` — sync dispatch разбивает аккаунты на 6 параллельных batch-worker'ов, **не** 32. Значение `APPLICATION_MAX_CONCURRENT_V8_ACTIONS=32` — это **общий ceiling** на ВСЕ одновременные V8 actions (sync workers + nested actions внутри них + cron-вызовы + другие пользовательские actions). На тяжёлых аккаунтах (1000+ баннеров) одного sync-worker'а достаточно для 60+ MiB carry-over; параллельно к нему могут работать nested actions и другие crons, что и заполняет ceiling.

**`convex/adAccounts.ts:upsertCampaignsBatch` line 1709** — последовательный цикл по `args.campaigns` внутри **одной internalMutation**. На batch=200 это 200 query+patch операций в одной транзакции — mutation держит весь read/write set до commit.

## 15. План оптимизации памяти Convex (отдельная сессия — код, не инфраструктура)

### Приоритет 1: sync per-account memory budget (`syncMetrics.ts`)

1. **Lightweight `getMtBanners`:** при обычном sync передавать минимально необходимый набор полей в VK API. Безопасный список (без nested field selection, которое VK API может не поддерживать):

   ```
   fields=id,campaign_id,textblocks,status,moderation_status
   ```

   **Самое тяжёлое — `content`** (превью с URL'ами, видео, креативы). Это поле точно убрать из обычного sync path. Если auto-link-video использует `content` — выносить эту логику в отдельную задачу/cron, не в основной sync.

   `textblocks` берём целиком (а не `textblocks.title.text`) — vk myTarget API не гарантированно поддерживает nested-field selection. `textblocks` сам по себе небольшой объект, в отличие от `content`.

2. **Scoped helper для тяжёлых данных:** вместо мутации `let banners = ...; ... ; banners = null;` — extract в helper-функцию, которая возвращает только компактный результат (Map или array of compact objects):

   ```typescript
   // Вместо императивной мутации:
   const bannerCampaignMap = await buildBannerCampaignMap(accessToken, accountId);

   async function buildBannerCampaignMap(token, id): Promise<Map<string,string>> {
     const banners = await getMtBanners(token, id);  // тяжёлый объект
     const map = new Map<string, string>();
     for (const b of banners) map.set(String(b.id), String(b.campaign_id));
     return map;  // banners выходит из scope, GC заберёт
   }
   ```

   Преимущества: scope явно ограничивает время жизни тяжёлой переменной, нет need в null-присваивании, чище для review.

3. **То же для `vkCampaigns`** → helper, возвращающий `groupData` Map напрямую.

4. **Streaming-обработка вместо «собрать всё → передать»:** разбить sync на этапы, каждый этап завершает работу с тяжёлой переменной до начала следующей.

### Приоритет 2: `upsertCampaignsBatch` chunk-size

- Замерить распределение размеров кампаний по аккаунтам **через Convex internal query** (admin-диагностика). SQL-доступ напрямую к `campaigns` не подходит: self-hosted Convex хранит app-таблицы во внутренних document/index структурах Postgres, не как обычную relational таблицу `public.campaigns`. Пример Convex-запроса:

  ```typescript
  // convex/admin.ts — добавить internalQuery
  export const campaignCountByAccount = internalQuery({
    handler: async (ctx) => {
      const all = await ctx.db.query("campaigns").collect();
      const counts: Record<string, number> = {};
      for (const c of all) counts[c.accountId] = (counts[c.accountId] || 0) + 1;
      return Object.entries(counts).sort((a, b) => b[1] - a[1]).slice(0, 20);
    },
  });
  ```

  Запустить через Convex Dashboard → Functions → run.

- Адаптивный chunk: для аккаунтов с >500 кампаний делать chunk 50, для остальных оставить 200.
- Альтернатива: разбить большую mutation на серию мелких через scheduler — потеряем атомарность, но снимем memory pressure.

### Чего НЕ делать

| Идея | Почему НЕ |
|---|---|
| `JSON.parse(JSON.stringify(x))` для разрыва ссылок | Удваивает пик памяти (создаётся вторая копия), не помогает с carry-over |
| Поднять `APPLICATION_MAX_CONCURRENT_V8_ACTIONS` 32 → 64 | Больше параллельных worker'ов с большими массивами = хуже, не лучше |
| Поднять Convex heap limit | Зашит в backend (96 MiB), env-override в self-hosted нет |

### Опционально для эксперимента (диагностика, не лечение)

- **Временно понизить** `APPLICATION_MAX_CONCURRENT_V8_ACTIONS` 32 → 16 в env. Все carry-over restart'ы валятся на `vkApi.js:getMtBanners` и `getMtStatistics` — те самые функции, которые планируется оптимизировать. Если при ALPP=16 restart'ы заметно сократятся — подтверждение гипотезы «проблема в одновременном удержании больших объектов параллельными worker'ами». Сам этот шаг **не лечение**, лечение — оптимизация функций.

### Верификация после деплоя оптимизации

1. **`docker logs --since 1h adpilot-convex-backend | grep TooMuchMemoryCarryOver | wc -l`** — целевое 0 или близко к 0.
2. **Замер WS на свежей вкладке `/dashboard` 1 час** — целевое 0–2 разрыва за час.
3. **Запустить sync на самом тяжёлом аккаунте вручную** через admin panel и наблюдать логи — Isolate не должен рестартовать.
4. **Контрольный замер `pg_stat_bgwriter`** — Postgres-метрики не должны деградировать (хотя оптимизация прямого влияния на Postgres не имеет).

### 15.2. Auto-link-video: сознательная временная регрессия

В primary sync удалён path, который читал `banner.content` и вызывал `internal.videos.autoLinkVideos`.

**Причина:** `content` — тяжёлое поле `getMtBanners`; именно такие full payloads были одним из подтверждённых источников Convex V8 memory carry-over restarts. Возврат `content` в обычный sync может вернуть WS 1006.

**Текущее состояние после memory-fix:**
- `internal.videos.autoLinkVideos` остаётся определённым, но не вызывается из primary sync.
- В `convex/syncMetrics.ts` добавлены `TODO(auto-link-cron)` в местах удаления.
- Memory entry должна быть обновлена во внешней project memory (`websocket-1006-investigation.md`): auto-link-video disabled in primary sync pending dedicated cron PR.

**Follow-up cron PR обязателен**, если авто-привязка видео должна продолжать работать. Требования зафиксированы в `docs/2026-05-03-convex-memory-carry-over-fix-design.md` раздел `8. Follow-up cron requirements`:
- частота 30–60 минут;
- videos-first bounded scan;
- лимит по candidates/banners, не по accounts;
- fair ordering по `lastLinkAttemptAt ASC NULLS FIRST`, fallback `_creationTime ASC`;
- группировка по `accountId` после bounded/fair выборки;
- per-account sequential, без `Promise.all(accountIds.map(...))`;
- scoped full `getMtBanners(fields: "id,content")`;
- zombie protection через TTL/attempt-counter;
- per-run cap и отдельная production verification.
